# Site Design Guide

[README](../README.md) | [CLI](cli.md) | [Authorship](authorship.md) | [Publishing](publishing.md) | [Python API](python-api.md) | [Reference](reference.md)

This guide covers site structure: `colophon.yml`, templates, routes, static files, vendor assets, images, and template helpers.

## Project Layout

Colophon sites are ordinary directories. The generator can live outside the site, and every command accepts `--config` so scripts and CI can point at a specific project.

```text
my-site/
  colophon.yml
  site_hooks.py
  content/
    site.yaml
    index.yml
    images.yml
    post-sidebar.yml
    pages/about.md
    posts/first-note.md
  templates/
    base.html
    index.html
    post.html
    simple.html
    archive.html
    tag.html
    feed.xml
  static/
    styles.css
    vendor/
  _site/
```

## `colophon.yml`

`colophon.yml` controls filesystem paths, trusted Python modules, and browser vendor assets.

```yaml
paths:
  content: content
  templates: templates
  static: static
  output: _site
  deploy: content/deploy.yaml
python:
  modules:
    - site_hooks.py
vendor:
  mode: auto
  local_dir: vendor
  required: []
  assets:
    webawesome:
      enabled: false
```

Paths are resolved relative to the config file unless absolute. `paths.deploy` is the only deploy-config path key.

Strict validation rejects aliases and shorthand forms such as `paths.project`, `paths.deploy_config`, `vendor.require`, scalar lists, boolean strings, and asset boolean shorthand.

## Site Config, Templates, and Routes

`content/site.yaml` defines site-wide data, template aliases, and route rules:

```yaml
site:
  title: Example Site
  description: A site generated by Colophon.
  url: https://example.test
  nav:
    - label: Home
      href: /

templates:
  default: index.html
  page: index.html
  post: post.html
  static: simple.html

routes:
  - match: /posts/**
    template: post
  - match: /**
    template: page
```

Template selection order:

1. `template` or `bindings.template` in the content file.
2. The most specific matching route rule in `content/site.yaml`.
3. `templates.default`.

Static pages under `content/pages/` default to the `static` template unless they set their own template.

`routes` accepts exact paths and prefix globs ending in `/**`.

## Templates

Templates receive page data as top-level variables plus helper globals. Common values:

| Name | Meaning |
| --- | --- |
| `site` | Data from `content/site.yaml` after defaults and expressions resolve. |
| `title`, `summary`, `date`, `tags` | Page or post values from frontmatter and defaults. |
| `article` | Rendered Markdown body for Markdown files. |
| `toc` | Level-two Markdown headings. |
| `page.route` | Current route, such as `/about/`. |
| `page.source_chain` | Content files merged to build the page. |
| `collections.posts` | Listed pages under `/posts/`, sorted by date descending. |
| `pages.all` | Listed page summaries for the whole site. |
| `assets` | Colocated source assets copied for this page. |

Useful globals and filters:

```jinja
{{ public_url(page.route) }}
{{ image("cover", "card").url }}
{{ vendor_url("webawesome", "webawesome.loader.js") }}
{{ vendor_enabled("mastodon-comments") }}
{{ date | date }}
{{ title | slugify }}
{{ "{title}" | fmt }}
```

See [Template reference](template-reference.md) for the complete template/global reference.

## Static Assets

Static files under `static/` are copied to the output root. For example:

- `static/styles.css` -> `/styles.css`
- `static/assets/demo.svg` -> `/assets/demo.svg`

Colocated non-content files beside a page source are copied and exposed in `assets`.

## Browser Vendor Assets

Vendor assets are configured in `colophon.yml` under `vendor`. Built-ins include `webawesome`, `dompurify`, `font-awesome`, `mastodon-comments`, and `mastodon-embed-timeline`.

```yaml
vendor:
  mode: auto
  local_dir: vendor
  required:
    - webawesome
  assets:
    dompurify:
      enabled: true
```

`vendor.mode`:

- `auto`: use local files when present, otherwise CDN URLs
- `cdn`: always use CDN URLs
- `local`: require local files and fail if missing

Templates should use `vendor_url(name, path)` instead of hardcoded CDN or local paths:

```jinja
<script defer src="{{ vendor_url('dompurify', 'purify.min.js') }}"></script>
```

Download selected assets:

```bash
colophon vendor download --config colophon.yml
colophon vendor download --config colophon.yml --asset webawesome --force
colophon vendor download --config colophon.yml --dry-run
```

## Images

Content images live under `content/images/` and are described by logical names in `content/images.yml`:

```yaml
images:
  demo_card:
    file: demo.png
    alt: Generated demo image
    width: 640
    height: 360
    fit: cover
    crop: auto
    variants:
      thumb:
        width: 320
        height: 180
```

Use the `image()` helper in templates:

```jinja
{% set card = image("demo_card", "thumb") %}
<img src="{{ card.url }}" alt="{{ card.alt }}" width="{{ card.width }}" height="{{ card.height }}">
```

Direct image paths are also supported:

```jinja
{{ image("/assets/demo.svg").url }}
{{ image("/images/original.png").url }}
{{ image("https://example.test/image.jpg").url }}
```

Missing logical image names, missing configured image files, and missing direct `/images/` or `/assets/` references fail the build with an asset error. Generated derivatives are written under `_site/images/generated/`.

## Design Philosophy

Colophon is intentionally opinionated:

- content-first: YAML and Markdown are the primary interface
- static output: the build writes files that can be served by cheap hosting
- strict builds over permissive builds: bad config, missing images, and unknown keys fail early
- trusted local Python over plugin packages: hooks are code in your site repository
- Mastodon and indie-web style publishing are built in
- no frontend bundler is required by default
